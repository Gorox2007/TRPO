# Практика: Сравнение RabbitMQ и Redis

## Что сделано

- Поднят стенд из 3 частей: `producer -> broker -> consumer`.
- Для обеих технологий используются одинаковые условия:
  - один и тот же формат сообщения (JSON с `sent_ts_ms` и `payload`);
  - одинаковая длительность прогона;
  - одинаковые параметры нагрузки.
  - одинаковые лимиты ресурсов контейнеров (`1 CPU`, `512 MB RAM`).
- Реализованы 3 обязательных эксперимента:
  - базовое сравнение;
  - влияние размера сообщения;
  - влияние интенсивности потока.
- Собираются обязательные метрики:
  - `messages/sec`;
  - средняя задержка;
  - `p95` и `max` задержка;
  - отправлено / обработано / ошибки / потери;
  - точка деградации `single instance`.

## Структура папки

- `benchmark.py` - скрипт всех обязательных прогонов.
- `docker-compose.yml` - `RabbitMQ` и `Redis`.
- `requirements.txt` - Python зависимости.
- `results/` - автоматически создаваемые результаты (`raw_results.csv`, `summary.md`).
- `screenshots/` - скриншоты для отчета.

## Запуск

```bash
cd broker_comparison_practice
python3 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
docker compose up -d
python benchmark.py --duration-sec 20 --base-rate 2000
```

Остановка стенда:

```bash
docker compose down
```

## Параметры экспериментов

- Размеры сообщений: `128 B`, `1 KB`, `10 KB`, `100 KB`.
- Интенсивность потока: `1000`, `5000`, `10000 msg/sec`.
- Длительность каждого прогона: `20 sec` (можно менять через `--duration-sec`).

### Выводы

1. Какой брокер показал большую пропускную способность: `Redis` (в base и на высоких rate обрабатывает заметно больше сообщений/сек).
2. Какой брокер лучше переносит рост размера сообщения: `Redis` (до `10 KB` почти без просадки, на `100 KB` просадка умеренная; у RabbitMQ резкая деградация).
3. При какой нагрузке `single instance RabbitMQ` начинает деградировать: с `5000 msg/sec`.
4. При какой нагрузке `single instance Redis` начинает деградировать: с `5000 msg/sec` по критерию `<95%` от target, при `10000 msg/sec` дополнительно резко растет latency.
5. Какой инструмент подходит лучше для этого сценария и почему: собственный `producer/consumer` на Python, потому что он дает полный контроль над payload, rate и сбором метрик (sent/processed/lost/latency) в одинаковой логике для обоих брокеров.
