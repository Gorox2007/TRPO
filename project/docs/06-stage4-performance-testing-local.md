# 6. Этап 4: Оптимизация, тесты и локальный запуск без Celery

В этой версии Stage 4 проект развивается без дополнительных сервисов. Архитектура остается монолитной:
- Telegram Bot API
- Python-приложение
- PostgreSQL

Celery, Redis и отдельные фоновые воркеры не используются.

## 6.1 Оптимизация производительности

Сделаны две практические оптимизации:
- при подготовке пачки рекомендаций бот выбирает `top-N` кандидатов без полной сортировки всего списка;
- в PostgreSQL добавлены составные индексы под самые частые запросы.
- текущее состояние просмотра кандидата хранится в БД, а не только в памяти процесса.

Изменения в коде:
- `stage2_bot/core.py`
  - `rank_candidates(...)` теперь умеет получать только нужное количество лучших кандидатов;
  - `/next` строит именно ту пачку, которая нужна для кэша, а не сортирует лишние элементы.
- `stage2_bot/db.py`
  - добавлено надежное хранение `current_candidate_user_id` через `user_view_state`;
  - Telegram username и имя пользователя обновляются на каждом сообщении, а не только на `/start`.

Изменения в БД:
- `idx_users_status_updated_at`
- `idx_users_city_gender`
- `idx_user_photos_active_lookup`
- `user_view_state`
- `users.telegram_chat_id`
- `users.last_seen_at`

## 6.2 Оптимизация базы данных

Stage 4 не меняет основную модель таблиц, но усиливает ее индексами:
- поиск активных кандидатов быстрее проходит по `status + updated_at`;
- выбор активных фотографий пользователя быстрее использует `user_id + is_active + position`.

Это безопасное улучшение для MVP: логика бота остается прежней, а тяжелые запросы выполняются эффективнее.

## 6.3 Тестирование продукта

Тесты сделаны без внешних библиотек, на стандартном `unittest`.

Покрыты сценарии:
- корректный порядок ранжирования;
- возврат только нужного количества лучших кандидатов;
- использование предкэшированной пачки кандидатов в `/next`.
- выдача контакта при взаимном лайке;
- показ списка мэтчей через `/matches`.

Запуск:

```bash
python3 -m unittest discover -s tests -p 'test_*.py'
```

## 6.4 Локальный запуск

Для демонстрации на зачете достаточно:

```bash
export TELEGRAM_BOT_TOKEN="your_bot_token"
export DATABASE_URL="postgresql://postgres:postgres@localhost:5432/dating_bot"
python3 -m stage2_bot.run
```

Если нужны тестовые данные:

```bash
psql postgresql://postgres:postgres@localhost:5432/dating_bot -f db/test_seed.sql
```

## 6.5 Почему без Celery

Эта реализация сознательно упрощена:
- меньше инфраструктуры;
- проще запускать и показывать локально;
- меньше точек отказа;
- требования Stage 4 закрываются за счет оптимизации, тестов и более надежного монолита.
